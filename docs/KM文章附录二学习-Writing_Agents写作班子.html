KM 文章附录二学习：Writing Agents 写作班子

> 原文标题：附录二：我的写作班子（Writing Agents）——三个 AI 角色 + 一个编排器 > 学习日期：2026-03-22 > 核心理念：有了规范，谁来执行？一个人扮三个角色，加一条流水线把它们串起来。

🎯 核心思想

为什么要拆角色？

人类写作的三种状态：

  1. 策划：想选题、搭结构
  2. 写手：埋头写正文
  3. 编辑：回头审稿、修改

AI 的问题：

  • ❌ AI 不会自动切换状态
  • ❌ 让它"写完之后自己改改"，结果是一边写一边改
  • ❌ 既没有写痛快，也没有改彻底

解决方案：

  • ✅ 拆开成三个独立角色
  • ✅ 各管各的阶段，谁也不越界
  • ✅ 用编排器（流水线）把它们串起来

👥 三个角色详解

📋 角色分工表

| 角色 | 负责阶段 | 一句话定位 | 核心原则 | |------|---------|-----------|----------| | 策划（Planner） | 系列规划 → 大纲 | 定结构，不写字 | 11 个 slot 全填满，12 维度控制信息完整 | | 写手（Writer） | 初稿 → 定稿 | 只管写，不管审 | 初稿追覆盖度，定稿按编辑反馈改 | | 编辑（Editor） | 深度分析 → 定稿审核 → 系列维护 | 只管审，不动笔 | 30+ 项检查清单逐项过，CRITICAL 不通过就是不通过 |

角色边界（非常硬）

| 角色 | 做什么 | 不做什么 | |------|-------|---------| | 策划 | 系列规划、大纲 | ❌ 不写正文（交出大纲，任务就结束） | | 写手 | 初稿、定稿 | ❌ 不做验收（写完交给编辑，编辑说改就改，不辩解） | | 编辑 | 深度分析、定稿审核、系列维护 | ❌ 不改文章（只出分析报告和检查清单，具体修改由写手执行） |

为什么边界要这么硬？ > 因为一旦角色边界模糊，AI 就会"自我和解"——写手写完觉得还行，编辑看了也觉得还行，最后质量就下去了。

📋 策划（Planner）：定结构，不写字

负责阶段

  1. 系列规划
  2. 大纲

系列规划阶段

用一个 11 slot 的模板引导填充：

  • 系列定位
  • 内容矩阵
  • 暗线设计
  • 阅读路径
  • ……

大纲阶段（两步走）

第一步：初稿态大纲（~200 行）

  • 章节骨架
  • 结构框架

第二步：控制态大纲（~400 行）

  • 12 维度控制信息
  • 每章预期字数
  • 核心立意
  • 素材分布
  • 风格标记
  • ……

控制态大纲是整个流水线最重要的中间产物：

  • 写手后面写初稿靠它
  • 编辑后面做深度分析靠它

✍️ 写手（Writer）：只管写，不管审

负责阶段

  1. 初稿
  2. 定稿

初稿阶段

工作方式：

  • 接过策划的控制态大纲
  • 逐章展开

核心原则： > "追求覆盖度，不追求完美"——先把素材铺满，别纠结措辞。

定稿阶段

工作方式：

  • 接过编辑的深度分析报告
  • 精修全文

工作模式：Evaluator-Optimizer 循环

写手精修 → 编辑评估 → 不通过？→ 写手再改 → 编辑再评 → ...（最多 3 轮）

反馈机制：

  • 每轮编辑都会给出结构化反馈：
- 哪些检查清单没过 - 具体问题在哪里 - 建议怎么改
  • 写手聚焦修正，不辩解，不讨价还价

📝 编辑（Editor）：只管审，不动笔

负责三件事

  1. 深度分析
  2. 定稿审核
  3. 系列维护

1. 深度分析（对初稿的全面体检）

四步流程

| 步骤 | 内容 | |------|------| | 交叉验证 | 核心立意和内容是否一致，数字前后是否矛盾 | | 三层输出 | 结构分析（章节权重）、内容分析（素材覆盖）、风格分析（情绪曲线） | | 噪音过滤 | 区分"必须改"和"可以改"，不让细节淹没核心问题 | | 输出报告 | 输出分析报告，交给写手 |

2. 定稿审核（全量检查清单逐项验收）

检查项总数：

  • 30+ 项通用检查清单
  • 6 项风格检查清单
  • 快速七问

每一项都是二值判断：Pass / Fail，不搞灰度。

质量分级

| 通过率 | 判定 | |--------|------| | ≥ 90% 且 CRITICAL 全过 | ✅ 直接通过 | | 80%~90%，无 CRITICAL 失败 | ⚠️ 标注风险点，提交你决定 | | < 80% 或 CRITICAL 失败 | ❌ 必须继续改 |

3. 系列维护

快速模式（每篇必做）

  • 更新概念索引
  • 去重
  • 暗线追踪

深度模式（每 3-5 篇）

  • 全系列一致性检查
  • 系列规划调整
  • 风格演化分析

隐藏职责：观察日志

每轮审核结束后写入观察日志：

  • 哪条检查清单反复失败
  • 哪条规则形同虚设
  • 哪种修改建议没被采纳

这些日志是下一次规范迭代的种子。

🔄 编排器：不是 AI，是流水线

关键理解

编排器不是 AI：

  • 它是一段预定义的代码路径
  • 阶段是固定的
  • 分工是明确的
  • 不需要 AI 来决定"下一步该做什么"

编排器做五件事

| 任务 | 说明 | |------|------| | 1. 确认三元组 | (阶段, 风格, 类型) → 决定加载哪些规范文件 | | 2. 按固定流水线分派任务 | 策划→写手→编辑→写手↔编辑循环 | | 3. Gate 检查 | 每个阶段完成后检查清单通过率，决定通过还是回退 | | 4. 在角色之间传递产出 | 控制上下文大小，每次只传阶段产出，不传完整历史 | | 5. 异常处理 | 重试 3 次仍然不过？提交你裁决 |

Gate 核心逻辑

检查清单通过率 < 80%      → 回退并列出未通过项
其余                      → 通过，进入下一阶段

完整流水线

② [策划] 大纲 → Gate→
③ [写手] 初稿 → 
④ [编辑] 深度分析 → 
⑤ [写手] 定稿 ↔ [编辑] 定稿审核 （循环，最多 3 轮）→ Gate→
⑥ [编辑] 系列维护

独立文章：跳过系列规划和系列维护，只走中间四步（②③④⑤）。

📊 四种协作级别（渐进式）

| 级别 | 做法 | 适用场景 | |------|------|----------| | Level 0 | 不分角色，规范当参考手册翻 | 刚开始写，规范还只有几条 | | Level 1 | 不分角色，但每个阶段手动切换 prompt | 规范有 10+ 条，开始感觉需要分工 | | Level 2 | 策划 + 写手合并，编辑独立 | 规范有 20+ 条，审稿复杂度上升 | | Level 3 | 三角色 + 编排器全流程 | 规范 28 个文件，系列写作，质量控制严格 |

你不需要一开始就搭三个角色 + 编排器。和规范一样，协作模式也是渐进的。

🎯 关键设计原则

1. 角色边界要硬

| 原则 | 说明 | 反例 | |------|------|------| | 策划不写正文 | 交出大纲就结束 | ❌ 策划顺手写了几段开场白 | | 写手不做验收 | 写完交给编辑 | ❌ 写手自己觉得还行就发布 | | 编辑不改文章 | 只出报告和检查结果 | ❌ 编辑直接改了几处 |

为什么要这么硬？ > 一旦边界模糊，AI 就会"自我和解"，质量就下去了。

2. Gate 检查不能软

Gate 是质量关卡：

  • CRITICAL 有一项未过 → 强制回退
  • 通过率 < 80% → 回退并列出未通过项
  • 不能因为"差不多"就放行

3. 编排器是代码，不是 AI

不要让 AI 决定流程：

  • ❌ "AI 你看看现在该做什么了"
  • ✅ 流程是固定的：策划→写手→编辑→循环

AI 只负责执行：

  • 策划 AI 负责出大纲
  • 写手 AI 负责写内容
  • 编辑 AI 负责审核
  • 编排器负责调度（代码）

4. 上下文传递要精简

不要传完整历史：

  • ❌ 把所有对话历史都传给下一个角色
  • ✅ 只传阶段产出（大纲、初稿、分析报告等）

控制 Token 成本：

  • 大纲阶段 → 只传系列规划（如果有）
  • 初稿阶段 → 只传控制态大纲
  • 定稿阶段 → 只传深度分析报告

5. 异常处理要明确

重试次数上限：

  • 每个阶段最多重试 3 次
  • 3 次后仍不过 → 提交人工裁决

不要无限循环：

  • ❌ 让 AI 反复改到通过为止
  • ✅ 3 次不过说明有系统性问题，需要人介入

阶段 3：编排器实现（2-3 个月）

目标：开发自动化流水线

编排器功能

  1. 确认三元组
   def confirm_context(style, type, stage):
       # style: xiaohongshu | webnovel
       # type: short-form | long-form | serial
       # stage: planning | outline | draft | final
       return load_modules(style, type, stage)

  1. 固定流水线
   def pipeline(content_type):
       if content_type == 'single':
           stages = ['outline', 'draft', 'analysis', 'final']
       elif content_type == 'series':
           stages = ['series_planning', 'outline', 'draft', 
                    'analysis', 'final', 'series_maintenance']
       
       for stage in stages:
           result = execute_stage(stage)
           if not gate_check(result):
               retry(stage, max_retries=3)

  1. Gate 检查
   def gate_check(result):
       if has_critical_failure(result):
           return False  # 强制回退
       if pass_rate(result) < 0.8:
           return False  # 回退
       return True  # 通过

  1. 产出传递
   def pass_to_next(stage_output):
       # 只传必要产出，不传完整历史
       if current_stage == 'outline':
           return {'outline': stage_output}
       elif current_stage == 'draft':
           return {'draft': stage_output, 'outline': outline}
       # ...

  1. 异常处理
   def retry(stage, max_retries=3):
       for i in range(max_retries):
           result = execute_stage(stage)
           if gate_check(result):
               return result
       
       # 3 次后仍不过，提交人工裁决
       return escalate_to_human(result)

🎓 学习要点总结

核心理念

  1. 角色分工要明确：策划、写手、编辑各司其职
  2. 边界要硬：谁也不越界，防止"自我和解"
  3. 编排器是代码：不需要 AI 决定流程
  4. Gate 检查不能软：CRITICAL 必须过，通过率必须 ≥ 80%
  5. 渐进式演进：不需要一开始就搭完整架构

实践建议
