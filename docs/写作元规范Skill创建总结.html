写作元规范 Skill 创建总结

> 创建时间: 2026-03-22 > 版本: v1.0.0 MVP > 状态: ✅ 已完成并可用

🎯 项目背景

基于 KM 文章《让 AI 像你一样写作》 的理念，为内容创作部门构建统一的写作元规范体系，让 AI 能够按照精确的标准进行内容创作。

核心目标：

  • 把"好文章应该长什么样"的标准，变成 AI 能理解、能执行、能自检的结构化描述
  • 支持多平台内容创作（小红书、网文、公众号、技术文章）
  • 采用渐进式迭代，从 MVP 开始，持续优化

📋 核心内容概览

通用规则（10 条）

🔴 CRITICAL（5 条）

  1. CR-01: 噪音原则 - 读者尚未建立认知的概念，提及即噪音
  2. CR-02: 每个元素服务于核心立意 - 文章的每一段都必须为核心观点服务
  3. CR-03: 改进的边界 - 明确告诉读者"什么能改进，什么改进不了"
  4. CR-04: 意象必须承载功能 - 比喻、类比去掉后，文章结构应该受损
  5. CR-05: 取舍是人的活 - AI 提供建议，人做决策

🟡 STANDARD（5 条）

  1. ST-01: 开篇三段定位 - 引起注意 + 明确主题 + 说明价值
  2. ST-02: 一段一个中心 - 避免混杂多个主题
  3. ST-03: 数据必须有来源 - 注明来源或测试条件
  4. ST-04: 结构对称 - 问题与方案的篇幅比例在 3:7 到 5:5
  5. ST-05: 衔接自然 - 段与段之间必须有自然过渡

小红书风格核心

  • 基调：轻松、亲切、真诚（用"你"，多短句，适度口语化）
  • 标题：6 种模式（数字型/疑问型/对比型/痛点型/福利型/情感型）
  • 结构：痛点/场景开场 → 分点展开 → 总结提炼 → 行动号召
  • Emoji：恰到好处（密度适中），语义一致，视觉平衡
  • 情感共鸣：痛点共鸣 + 场景共鸣 + 成就共鸣

网文风格核心

  • 悬念设置：
- 章节悬念（1-2 章）：推动读者点"下一章" - 卷级悬念（5-10 章）：维持长期兴趣 - 全书悬念（全文）：核心驱动力
  • 爽感节奏：
- 小爽点：每 1-2 章（打脸/逆袭/揭秘/复仇/获得） - 中爽点：每 5-8 章 - 大爽点：每 15-20 章

  • 人物塑造：
- 主角：有目标，有成长，行为有动机 - 配角：有功能，有层次，不脸谱化 - 反派：有智商，有动机，有威胁

  • 节奏控制：紧张 40% + 铺垫 40% + 过渡 20%

🎯 核心设计理念

1. 从实践中来

> "不要一开始就设计规范。先写 3 篇，记录所有审稿意见，反复出现的意见 = 你的隐式规则。"

2. 规则必须可验证

  • ❌ "写得好" → ✅ "核心内容覆盖 ≥ 80%"
  • ❌ "结构合理" → ✅ "核心章占比最大，非均分"

3. 正反例比规则更重要

> "规则只说'应该'，但'应该'的边界在哪里，只有正反例能展示。"

4. 不追求完美，追求可迭代

> "第一版可以只有 5 条规则和 3 条 CL。第九版才会有 28 个文件。但每一版都比上一版好——这就够了。"

5. 人的决策不可替代

> "AI 提供数据和建议。你做判断和取舍。这件事不能反过来。"

🎉 项目亮点

1. 架构设计

  • 六层框架 清晰分离关注点（通用/风格/策略/元/理论）
  • 三层叠加 灵活组合，支持多场景
  • 路由器机制 自动确定加载内容

2. 规则体系

  • 分级管理 (CRITICAL/STANDARD/RECOMMENDED) 突出重点
  • 可验证性 每条规则都有明确的检查标准
  • 理论支撑 规则来源有据可查

3. 实用性

  • 检查清单 3 套通用清单 + 风格特定清单，覆盖全流程
  • 手法库 8 条通用手法，按阶段分组，可操作性强
  • 正反对照 每个概念都有示例说明

4. 可扩展性

  • 风格包 插拔式设计，易于添加新风格
  • 策略库 预留类型策略扩展空间
  • 迭代机制 明确的收敛标准和迭代流程

📝 使用建议

首次使用

  1. 先读 SKILL.md 理解整体架构（15 分钟）
  2. 根据要写的内容类型，选择对应场景
  3. 按场景建议的顺序阅读相关文件
  4. 边读边实践，不要死记硬背

日常使用

  1. 大纲阶段：快速复习 shared/rules.md + 对应风格的 definition.md
  2. 初稿阶段：参考 shared/techniques.md 中的初稿手法
  3. 定稿阶段：使用 shared/checklists.md + 对应风格的 checklist.md

迭代维护

  1. 每完成 3-5 篇内容，进行一次结构性迭代
  2. 记录反复出现的问题，考虑是否需要新增规则或手法
  3. 更新 meta/changelog.md 记录变更
